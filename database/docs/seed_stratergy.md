# Seed Strategy

## Purpose

This document defines how database seed data is generated, loaded, validated, and maintained across the Multi-Agent Customer Support System.

It establishes:

* Which tables are manually seeded
* Which tables are system-generated
* How customer stories map to database records
* Seed execution order
* Seed validation rules

This document is the authoritative source of truth for all database seeding activities.

---

# Seeding Philosophy

The system follows a strict Input → Processing → Output architecture.

Business entities are manually seeded.

Agent-generated entities are created by workflow execution.

```text
Customer Story
      ↓
Seeded Tables
      ↓
Agent Execution
      ↓
Generated Tables
```

The database must never contain generated records that cannot be traced back to a valid customer story.

# Seed Execution Order

The master seed must execute in the following order.

1. customers.sql
2. auth_accounts.sql
3. subscriptions.sql
4. billing_history.sql
5. orders.sql
6. tickets.sql
7. feedback_signals.sql

This order guarantees foreign key integrity.

---

# Master Seed Script

File:

```text
database/seeds/master/seed_master.sql
```

Execution Flow:

```sql
BEGIN;

\i customers.sql

\i auth_accounts.sql

\i subscriptions.sql

\i billing_history.sql

\i orders.sql

\i tickets.sql

\i feedback_signals.sql

COMMIT;
```

# Validation Rules

## Rule 1

Every seeded record must belong to a documented customer story.

---

## Rule 2

No orphan foreign keys are allowed.

---

## Rule 3

Generated tables must never be manually seeded.

---

## Rule 4

All generated records must be reproducible through workflow execution.

---

## Rule 5

Customer stories are the single source of truth for seed generation.

---

# Generated Table Protection Rule

The following tables are system-generated artifacts and must never be manually seeded:

* customer_support_profiles
* churn_alerts
* refund_workflows
* refund_audit
* processed_refunds
* escalation_cases
* escalation_audit
* escalation_feedback
* notification_outbox
* ticket_transcripts
* translation_records
* account_security_audit
* crm_events
* processed_events
* active_workflows
* idempotency_keys
* proactive_events
* proactive_outreach_registry

These tables are outputs of workflow execution.

If data is manually inserted into these tables, the integrity guarantees established by the Customer Stories, Table Ownership Matrix, and Seed Strategy are violated.

Generated tables must be populated exclusively through agent execution.


# Future Expansion Policy

When a new agent is introduced:

1. Add a customer story.
2. Update table ownership if required.
3. Update story-to-table mapping.
4. Update seed strategy.
5. Add integration tests.

No new seed data may be added without a corresponding customer story.
